# 25 — Policy handbook generator from a tag

**What to build:** The last-mile attempt's mechanical half (PRD §6.8): an always-in-sync, human-readable policy handbook generated from a signed policy tag — every policy, its version, its enforcement tier, its "why" — so the operational manual can never drift from enforced policy. Regenerable in CI from any tag.

**Blocked by:** 04 — Signed release pipeline, 22 — Advisory metadata schema.

**Status:** done

- [x] Running the generator against a tag produces a readable handbook covering every policy in that version
- [x] Regenerating from a different tag reflects that version's policy set — nothing hand-maintained
- [x] P3 acceptance holds: "the handbook regenerates from a tag"

## Comments

Done 2026-07-15. `policy` repo, `handbook/generate.sh <tag>`: reads every field (version,
enforcement action from `validationActions`, advisory metadata, full rationale) straight out of
`git show`/`git ls-tree` on the given tag's tree — no working-tree checkout, no hand-maintained
list of policies.

Proven against two real, already-signed tags: `v1.0.1` renders 2 policies, `require-department-label`
shown correctly at `Audit` (its launch enforcement, pre-promotion) with `Created`/`Last reviewed`
gracefully blank (that tag predates issue 22's annotation schema — the field is read, not
fabricated, when it doesn't exist). `v2.1.1` renders 3 policies, correctly picking up
`require-owner-annotation` which didn't exist at `v1.0.1`. Same script, same command shape,
different output purely because the tag's tree differs — nothing regenerated by hand.

No generated `HANDBOOK.md` is committed to the repo: doing so would itself be exactly the
drift-the-moment-the-next-tag-lands problem this ticket exists to avoid. README documents the
command instead.
